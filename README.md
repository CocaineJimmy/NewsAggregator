## 1. Описание проекта

**Минск.Новости** — это многостраничная новостная платформа с полным стеком (React + Express.js + PostgreSQL), ориентированная на новости Минска и Беларуси. Приложение позволяет читателям просматривать материалы по категориям, комментировать статьи, сохранять публикации в закладки и получать XP-очки за активность на платформе. Администраторы управляют контентом и пользователями через встроенную панель.

Дизайн платформы вдохновлён ведущими медиа-сервисами — Medium, Apple News и Duolingo: читаемая типографика, минималистичный интерфейс, тёмная/светлая тема.

**Основная цель:** демонстрация полноценного веб-приложения уровня production для дипломной работы, включающего аутентификацию, базу данных, RESTful API, геймификацию и адаптивный интерфейс.

### Слои архитектуры

**Фронтенд (client/src/)**
Одностраничное React-приложение (SPA). Роутинг на стороне клиента с помощью Wouter. Все данные fetching-а инкапсулированы в хуках TanStack Query с кэшированием. Аутентификация через React Context (AuthContext). UI построен на основе shadcn/ui — готовые доступные компоненты.

**Бэкенд (server/)**
Express.js-сервер, обрабатывающий REST API-запросы и раздающий собранные статические файлы фронтенда в production. Бизнес-логика разделена между роутами (routes.ts) и слоем хранилища (storage.ts). Все записи в БД проходят через единый интерфейс IStorage.

**Общая типизация (shared/)**
Схема базы данных определена один раз в shared/schema.ts и используется как фронтендом (типы), так и бэкендом (ORM + Zod-валидация). Это исключает рассинхронизацию типов между слоями.

## 7. Система геймификации

Платформа реализует систему геймификации в стиле Duolingo, стимулирующую читательскую активность.

### Механика начисления XP

| Действие | XP |
|---|---|
| Первый просмотр статьи | +5 XP |
| Написание комментария | +10 XP |

**Важно:** XP начисляется только за **первый** просмотр каждой статьи. Повторные визиты не дают очков. Это обеспечивается таблицей `news_views` с уникальным ограничением `UNIQUE(user_id, news_id)`.

### Система уровней

Уровень пользователя вычисляется на основе накопленного XP:

```
Уровень = floor(XP / 100) + 1
```

| Уровень | Порог XP |
|---|---|
| 1 | 0–99 |
| 2 | 100–199 |
| 3 | 200–299 |
| N | (N-1)×100 – N×100-1 |

### Отображение прогресса

На странице профиля пользователь видит:
- Текущий уровень и прогресс-бар до следующего уровня
- Общее количество накопленного XP
- Количество написанных комментариев
- Количество сохранённых закладок

### Предотвращение читинга

При каждом запросе `GET /api/news/:id` сервер проверяет, просматривал ли пользователь эту статью ранее (`storage.hasUserViewedNews`). Если нет — записывает просмотр и начисляет XP. Уникальное ограничение на уровне БД гарантирует целостность данных даже при параллельных запросах.

## 8. Система безопасности

### Аутентификация

Платформа использует **сессионную аутентификацию** (не JWT). После успешного входа сервер создаёт сессию и отправляет браузеру HTTP-only cookie. При каждом последующем запросе cookie автоматически передаётся браузером.

```
Клиент                    Сервер                      БД
   │  POST /api/login       │                           │
   │  {email, password}     │                           │
   │──────────────────────►│                           │
   │                        │  getUserByEmail()         │
   │                        │──────────────────────────►│
   │                        │◄──────────────────────────│
   │                        │  bcrypt.compare()         │
   │                        │  session.userId = id      │
   │◄──────────────────────│  Set-Cookie: session=...  │
   │  200 OK + user data    │                           │
   │                        │                           │
   │  GET /api/me           │                           │
   │  Cookie: session=...   │                           │
   │──────────────────────►│                           │
   │                        │  req.session.userId       │
   │◄──────────────────────│                           │
   │  200 OK + user data    │                           │
```

### Хэширование паролей

Пароли хранятся в хэшированном виде с использованием **bcrypt**. Открытый текст пароля никогда не записывается в базу данных. При входе хэш проверяется функцией `bcrypt.compare()`.

### Защита маршрутов

Все защищённые API-маршруты используют middleware:

```typescript
// Проверка авторизации
function requireAuth(req, res, next) {
  if (!req.session.userId) {
    return res.status(401).json({ error: "Требуется авторизация" });
  }
  next();
}

// Проверка роли администратора
async function requireAdmin(req, res, next) {
  const user = await storage.getUser(req.session.userId);
  if (!user?.isAdmin) {
    return res.status(403).json({ error: "Доступ запрещён" });
  }
  next();
}
```

### Параметры безопасности сессий

| Параметр | Значение | Назначение |
|---|---|---|
| `httpOnly: true` | Включён | Защита от XSS (JavaScript не читает cookie) |
| `secure: true` | В production | Передача только по HTTPS |
| `maxAge` | 30 дней | Время жизни сессии |
| `secret` | `SESSION_SECRET` из .env | Криптографическая подпись cookie |
| Хранилище | PostgreSQL | Сессии хранятся в БД, а не в памяти |

### Управление пользователями

Администраторы могут блокировать аккаунты пользователей (`is_blocked = true`). При входе заблокированного пользователя сервер возвращает ошибку 403 до создания сессии. Ответы API никогда не включают поле `password`.

### Безопасность загрузки файлов

Multer настроен с ограничениями:
- Максимальный размер файла: **5 МБ**
- Допустимые MIME-типы: `jpeg`, `jpg`, `png`, `gif`, `webp`
- Имена файлов генерируются случайно (timestamp + random) для предотвращения перезаписи

## 10. Установка и запуск

### Требования

- Node.js 20+
- npm 10+
- Аккаунт [Neon](https://neon.tech) (бесплатный тарифный план достаточен)

### Переменные окружения

Создайте файл `.env` в корне проекта:

```env
# Строка подключения к PostgreSQL (получить в консоли Neon)
DATABASE_URL=postgresql://user:password@host/dbname?sslmode=require

# Произвольная строка длиной 32+ символа для подписи сессий
SESSION_SECRET=your_very_long_random_secret_string_here
```

> **Важно:** Никогда не коммитьте `.env` в репозиторий. Добавьте `.env` в `.gitignore`.

### Установка зависимостей

```bash
npm install
```

### Инициализация базы данных

```bash
# Применить схему (создать таблицы)
npm run db:push

# Заполнить БД начальными данными (admin, категории)
npx tsx server/seed.ts

# Опционально: заполнить 21 статьёй о Минске
npx tsx server/seed-articles.ts
```

После `seed.ts` в консоли будут выведены учётные данные администратора.

### Запуск в режиме разработки

```bash
npm run dev
```

Приложение запускается на `http://localhost:5000`. Vite обеспечивает горячую перезагрузку (HMR) фронтенда. Бэкенд перезапускается автоматически при изменении файлов.

### Сборка для production

```bash
npm run build
```

Команда создаёт:
- `dist/public/` — собранный фронтенд (Vite)
- `dist/index.js` — бандл бэкенда (esbuild)

### Запуск в production

```bash
npm start
```

Express-сервер раздаёт фронтенд как статические файлы и обслуживает REST API.

### Доступные npm-скрипты

| Скрипт | Описание |
|---|---|
| `npm run dev` | Запуск в режиме разработки (HMR + tsx) |
| `npm run build` | Сборка для production |
| `npm start` | Запуск production-сборки |
| `npm run db:push` | Синхронизировать схему БД с Drizzle |
| `npm run db:studio` | Открыть Drizzle Studio (GUI для БД) |
